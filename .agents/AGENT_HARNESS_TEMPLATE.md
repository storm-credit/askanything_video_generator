# Agent Expertise Harness Template

이 템플릿은 AskAnything 오케스트라의 모든 전문가 에이전트를 같은 깊이로 설계하기 위한 공통 골격이다.

## Harness Fields

### 1. Role Contract
- 에이전트가 맡는 전문 영역을 한 문장으로 정의한다.
- 다른 에이전트와 겹치는 책임을 명확히 분리한다.
- 최종 사용자가 체감하는 품질 목표를 포함한다.

### 2. Inputs
- 에이전트가 판단에 사용하는 코드/데이터/설정/생성물을 명시한다.
- 필수 입력과 선택 입력을 구분한다.
- 입력이 없거나 불완전할 때의 fallback 동작을 쓴다.

### 3. Expert Judgment Criteria
- 좋은 결과와 나쁜 결과를 구분하는 전문 기준을 적는다.
- 채널별, 포맷별, 언어별 차이가 있으면 별도 규칙으로 분리한다.
- 단순 취향이 아니라 관찰 가능한 조건으로 쓴다.

### 4. Hard Fail
- 발견 즉시 중단하거나 재작성해야 하는 조건이다.
- 가능하면 코드 검증 또는 정규식/스키마로 자동 확인 가능하게 쓴다.
- 하드 실패와 경고를 섞지 않는다.

### 5. Auto-Fix Policy
- 에이전트가 자동 수정해도 되는 범위를 적는다.
- 절대 건드리면 안 되는 보존 영역을 적는다.
- 수정 후 재검증 기준을 명시한다.

### 6. Output Contract
- 사람이 읽는 리포트인지, JSON인지, 수정된 cut 배열인지 고정한다.
- 필수 필드, 상태값, 실패 메시지 형식을 명시한다.
- 다음 에이전트가 소비하는 값은 이름까지 고정한다.

### 7. Code Wiring
- 실제 연결 파일과 함수/클래스를 적는다.
- 문서상 전문가와 런타임 Agent가 다르면 둘 다 적는다.
- 현재 미구현이면 `not wired`로 표시한다.

### 8. Verification Harness
- 스모크 테스트, 샘플 입력, 기대 결과를 적는다.
- 좋은 예, 나쁜 예, 경계 예를 최소 1개씩 둔다.
- 회귀 테스트로 바꿀 수 있는 항목을 따로 표시한다.

## Scoring Rubric

| Score | Meaning |
|---:|---|
| 9-10 | 전문가 롤, 판단 기준, 실패 조건, 자동수정, 코드 연결, 테스트가 모두 깊게 연결됨 |
| 7-8 | 전문 기준은 충분하지만 일부 출력/테스트/코드 연결이 약함 |
| 5-6 | 역할과 체크리스트는 있으나 판단 기준이나 자동수정 정책이 얕음 |
| 3-4 | 이름과 담당 파일 중심, 전문 기준이 거의 없음 |
| 1-2 | 문서/코드 연결이 불명확하거나 실제 에이전트로 보기 어려움 |
